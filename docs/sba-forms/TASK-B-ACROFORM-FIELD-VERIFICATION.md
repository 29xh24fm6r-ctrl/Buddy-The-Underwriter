## 2. What is confirmed instead: the current field names are self-admittedly not real AcroForm names

Every `fields.ts` file for the four wired forms says this directly in its
own header comment:

> `pdfCoord` is a rendering hint only... refined by banker QA once the
> official PDF template exists (ARC-00 Phase 0.C is currently blocked on
> network access to sba.gov; no coordinates have been mapped against a
> real template yet, so this field omits `pdfCoord` entirely rather than
> guess at placement).

The `key` values in `FORM_1919_SECTION_I_FIELDS` etc.
(`applicant_legal_name`, `full_name`, `ssn_last4`, ...) are **internal
semantic keys**, not real AcroForm field names — they were never claimed
to be. `render.ts` flattens them (e.g. `section_i.applicant_legal_name`)
and calls `form.getTextField(key)` against whatever the real template's
AcroForm actually contains.

## 3. The concrete risk this creates (confirmed, not speculative)

`render.ts`'s fill loop is written to **fail closed but silently**:

```ts
for (const [key, value] of Object.entries(values)) {
  try {
    const field = form.getTextField(key);
    field.setText(value);
  } catch {
    // Field name doesn't match this template's AcroForm — expected
    // until real coordinates/names are mapped against the ingested
    // PDF. Skip rather than fail the whole render.
  }
}
```

Real SBA/IRS AcroForm field names are essentially never semantic
snake_case like `applicant_legal_name` — Adobe-authored government forms
typically use autogenerated paths like
`topmostSubform[0].Page1[0].something[0]`. That means once a real template
*is* uploaded to `bank_document_templates`, the near-certain outcome is
that **every single `getTextField(key)` call throws, every field is
silently skipped, and `render.ts` returns `{ ok: true, pdfBytes }` for a
PDF that looks like the untouched blank original** — no error, no failing
test, nothing to catch this short of opening the PDF and looking.

This is inactive today only because `bank_document_templates` has 0 rows
(so every render call returns `TEMPLATE_NOT_AVAILABLE` first) — it is not
a defect in the current build, it's exactly the gap this task exists to
close before that changes. The moment someone uploads a real template,
this becomes live and silent.

**Not fixed in this pass** — per the task's own instruction, mismatches
get fixed as their own reviewable PRs once real field names are known, not
guessed at here.
