## Context

T0 went looking for the "equity seasoning + debt schedule auto-builder" the roadmap's Phase 3 (S4) entry mentions, expecting it to be document-extraction-driven (OCR off uploaded bank statements). It's real and tested, but not what was assumed:

- `src/lib/financialFacts/debtScheduleAutoBuilder.ts` exports `buildDebtSchedule(transactions: BorrowerBankTransactionLike[])` — a pure function that infers existing debt obligations (mortgage/credit-card/MCA/etc.) from **Plaid bank-transaction data**, per `specs/sba-30min-package/SPEC-S4-credit-pull-and-irs.md`. Fully unit-tested (6 cases).
- **It has zero callers anywhere in the codebase outside its own test file.** Not in a route, not in a job processor, not in the Brokerage concierge, not in any Underwriter cockpit flow.
- `src/app/api/deals/[dealId]/existing-debt/route.ts` provides GET/POST against a `deal_existing_debt_schedule` table, but it's gated by `ensureDealBankAccess` — a banker-side manual-entry API for the Underwriter cockpit, unreachable by a Brokerage borrower.
- `BORROWER_FIELD_REGISTRY` has no `existing_debt`/`debt_schedule` entries at all, and the concierge extraction prompt doesn't ask about it.

**Net: this is not "Brokerage is missing a field." No path anywhere in this product — Brokerage or Underwriter, conversational, Plaid-driven, or manual-by-borrower — gets a borrower's existing business debt into the system.** SBA underwriting requires a full existing-debt schedule (not just the debt being refinanced) for DSCR/global-cash-flow purposes. Every deal that's gone through this product to date has had this gap.

## Scope

1. **Open question to resolve first:** does Brokerage have a live Plaid connection today? **Answered: no** — no Plaid Link integration exists anywhere under `src/app/(borrower)/` or `src/lib/brokerage/`. Standing one up is genuinely gated on Plaid vendor credentials (same "environmental blocker, not a code gap" category as the Persona/DocuSeal/CAIVRS/SAM/E-Tran credentials ARC-00 flagged as never provisioned in this environment — see `docs/build-logs/ARC00_VENDOR_PROVISIONING_CHECKLIST.md`). Not attempted this pass; **not** blocking on the rest.
2. ~~Wire `buildDebtSchedule()` to run against real Plaid transactions~~ — **deferred, needs Plaid credentials.** What *is* done: `debtScheduleEntryToRow()` (`src/lib/financialFacts/existingDebtSchedule.ts`) maps a `DebtScheduleEntry` straight onto `deal_existing_debt_schedule`'s row shape, tagged `source='plaid_auto'` with its `confidence` carried through — when Plaid is wired, a job calls `buildDebtSchedule()` then this adapter then the same `insertExistingDebtScheduleEntry()` every other source already uses. No new table, no second migration.
3. **Persist into `deal_existing_debt_schedule` — done**, via a new shared module (`src/lib/financialFacts/existingDebtSchedule.ts` pure logic + `existingDebtScheduleWriter.ts` DB layer) both the existing banker route (`/api/deals/[dealId]/existing-debt`, now refactored to use it) and a new Brokerage borrower-facing route (`/api/brokerage/deals/[dealId]/existing-debt`, session-authenticated) call. An additive migration (`20260714_existing_debt_schedule_source_columns.sql`) added `source` (`manual_banker`/`manual_borrower`/`plaid_auto`) and `confidence` columns so all three producers share one table without collision.
4. **Structured fallback capture — done, as a standalone card**, not folded into Ticket 3 (which hasn't landed yet): `ExistingDebtCard.tsx`, mounted in `StartConciergeClient.tsx` under the existing captured-facts panel. Add/list/remove existing debts, plus an explicit "I don't have any other business debt" confirmation (writes an explicit `ADS=0` fact rather than leaving the denominator silently blank — many businesses genuinely have zero existing debt, and that's a different, valid state from "not entered yet").
5. **Downstream consumers — checked, and this turned out to be worse than one gap:** `computeTotalDebtService.ts` (Underwriter-cockpit DSCR pipeline) already reads `deal_existing_debt_schedule` directly — no bridge needed there. But the pipeline that actually matters for Brokerage — the SBA package/forward model (`sbaAssumptionsPrefill.ts` → `sbaForwardModelBuilder.ts`) — reads existing debt from a **third, disconnected representation**: a bare scalar `ADS` fact in `deal_financial_facts`, fabricated into a single-item placeholder array (`loanImpact.existingDebt`) with a hardcoded 60-month term and $0 balance. Worse, once an assumptions row exists, `sbaAssumptionsBootstrap.ts`'s own merge logic (`exLI?.existingDebt ?? prefillLI?.existingDebt ?? []`) means an empty `[]` stored on that row is never replaced by a later prefill re-run — nullish coalescing doesn't treat `[]` as nullish. Built `syncExistingDebtScheduleToDownstream()` to bridge both: writes the real aggregate to the `ADS` fact (respecting the same "document fact wins" precedence `propagateBorrowerFacts.ts` uses for every other concierge-sourced key), and replaces `loan_impact.existingDebt` with real per-creditor line items whenever it's still empty or exactly that fabricated placeholder — never when it holds real edited data. Called after every write on both the borrower and banker routes.
