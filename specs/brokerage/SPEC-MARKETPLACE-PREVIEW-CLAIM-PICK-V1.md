## Mission

Specify the marketplace state machine — preview → claim → borrower pick →
atomic unlock — deterministically enough that the next Claude Code pass
can build it without reasoning about timing edge cases or race
conditions.

This spec assumes the borrower intake + sealing slice from
SPEC-BROKERAGE-PRODUCTIONIZATION-V1 is already live and the Stage A RLS
hardening from SPEC-BROKERAGE-RLS-HARDENING is complete.

---

## Listing state machine

`marketplace_listings.status` transitions only along these edges:

```
pending_preview  --(scheduler @ 9am CT next BD)-->  previewing
previewing       --(scheduler @ 9am CT next BD)-->  claiming
claiming         --(3 claims placed)----------->    awaiting_borrower_pick
claiming         --(window closes, 0 claims)--->    expired
claiming         --(window closes, >0 claims)-->    awaiting_borrower_pick
awaiting_pick    --(borrower picks)------------>    picked
awaiting_pick    --(grace expires)------------->    expired   [P1]
expired          --(ops re-list)--------------->    relisted
relisted         --(scheduler @ next 9am)----->     previewing
picked           --(no transition out)
```

All edges except `borrower picks` are scheduler-owned. The scheduler is a
CRON-driven task that runs every 5 minutes and walks listings whose next
scheduled timestamp has passed.

---

## Preview window

- Listing.preview_opens_at = next business day 09:00 America/Chicago
  after `buddy_sealed_packages.sealed_at`.
- Listing.claim_opens_at = preview_opens_at + 1 business day @ 09:00 CT.
- Listing.claim_closes_at = claim_opens_at + 48 hours.

Business-day rule: skip Sat / Sun / US federal holidays. Holidays come
from a static `marketplace_holidays_us` table populated for the current
calendar year — ops refreshes annually.

Scheduler invariant: a listing never advances past a scheduled time it
has not yet reached, even if multiple scheduler runs overlap.

---

## Matching

`marketplace_listings.matched_lender_bank_ids` is computed at the moment
the listing is created (sealing → list). The match engine is the
existing `src/lib/brokerage/matchLenders.ts` — score band + geography +
NAICS prefix + min_dscr + sba_only.

Once matched, the set is frozen for the life of the listing. A re-list
recomputes from the current rate card + lender programs.

## Claim window

Concurrency rule: at any point in time, at most three rows exist in
`marketplace_lender_claims` with `status = 'claimed'` for a given
`listing_id`. The 4th claim attempt returns `claim_cap_reached`.

Atomicity rule: the cap is enforced inside a single transaction with a
row-level lock on the listing. Reference implementation in
`supabase/migrations/20260620000000_marketplace_claims_and_picks.sql`
(branch `claude/buddy-brokerage-product-CjK0a`) — function
`claim_marketplace_listing(listing, lender, user)` returns the new
claim id and flips listing to `awaiting_borrower_pick` when the cap is
reached.

Claim state transitions: `claimed → won | lost | relinquished | expired`.
`declined` is a separate terminal state for lenders who pass without
claiming; it does NOT count against the cap.

Lender RLS: a lender sees only its own rows in
`marketplace_lender_claims`. Brokerage ops sees all rows.

## Borrower pick

Pre-conditions:

- listing.status in (`claiming`, `awaiting_borrower_pick`).
- at least one row exists in `marketplace_lender_claims` for the listing
  with status='claimed'.
- the requesting cookie maps to a borrower session whose `deal_id`
  equals the listing's `deal_id`.

Operation: `pick_marketplace_winner(listing_id, winning_lender_bank_id)`
runs in a single transaction with FOR UPDATE on the listing:

1. Mark all other `status='claimed'` rows as `lost` with
   `decided_reason='borrower picked another lender'`.
2. Mark the winner's row as `won`.
3. Update listing.status to `picked`, listing.picked_at to now().
4. Insert a `marketplace_borrower_picks` row.

The function returns the new pick id. Caller verifies borrower ownership
of the deal BEFORE invoking — the RPC does not re-verify because it
runs SECURITY DEFINER.

## Atomic unlock

The borrower pick triggers three downstream releases — each is the
**read side** of the existing RLS policies, not a separate write:

1. **Borrower trident unlock.** Borrower portal serves
   `buddy_trident_bundles.mode = 'final'` for this deal. The bundle is
   generated by the existing `generateTridentBundle` workflow; the
   portal already gates on the existing `marketplace_borrower_picks`
   row to flip from preview to final.

2. **Winning lender package unlock.** Lender RLS policy on
   `deal_documents` / `buddy_sealed_packages` / `document_artifacts`
   filters using:

   ```sql
   exists (
     select 1
       from public.marketplace_borrower_picks p
       join public.bank_user_memberships m
         on m.bank_id = p.winning_lender_bank_id
     where p.deal_id = deal_documents.deal_id
       and p.reverted_at is null
       and m.user_id = auth.uid()
   )
   ```

3. **Losing lender access revocation.** Lender RLS on
   `marketplace_listings` is `matched_lender_bank_ids` membership; once
   listing.status = `picked` the policy adds `AND (lender = winner OR
   listing.status NOT IN ('picked'))`. The redacted KFS row remains
   visible for history; the full deal is not.

Audit: every transition writes one row to `ai_events` with `scope =
'marketplace'` and the deal_id. The pick row's `id` is the audit
correlator.

## Signed URLs

Trident PDF / XLSX downloads are served via short-lived signed URLs
minted at request time by `/api/portal/[token]/trident/download/[kind]`
(borrower) or `/api/lender/deals/[dealId]/trident/[kind]` (lender, won
deals only). TTL = 5 minutes.

The signed URL minter is the only place that consults
`marketplace_borrower_picks` to decide which mode to serve. No URL is
persisted.

## Cron / scheduler

Job: `/api/cron/brokerage/marketplace-tick` — runs every 5 minutes via
Vercel cron.

Body:

1. Find listings where `status = 'pending_preview'` and
   `preview_opens_at <= now()`. Flip to `previewing`.
2. Find listings where `status = 'previewing'` and `claim_opens_at <=
   now()`. Flip to `claiming`.
3. Find listings where `status = 'claiming'` and `claim_closes_at <=
   now()`. If 0 claims, flip to `expired`. Otherwise flip to
   `awaiting_borrower_pick`.
4. (P1) Find listings where `status = 'awaiting_borrower_pick'` and
   `picked_at IS NULL` and `claim_closes_at + 72h <= now()`. Flip to
   `expired` and notify ops.

Failures: scheduler is idempotent. A retry of step 3 on an already-flipped
listing is a no-op due to status guard.

## Audit log

Required `ai_events` rows for each marketplace transition:

| Event | scope | action | input_json | output_json |
| --- | --- | --- | --- | --- |
| listing previewing | marketplace | preview_open | listing_id | matched_count |
| listing claiming | marketplace | claim_open | listing_id | {claim_closes_at} |
| lender claim | marketplace | lender_claim | {listing_id, lender_bank_id} | {claim_id, active_count} |
| listing cap reached | marketplace | claim_cap_reached | listing_id | {claim_ids[]} |
| borrower pick | marketplace | pick | {listing_id, winning_lender_bank_id} | {pick_id, losing_lender_bank_ids[]} |
| listing expired | marketplace | listing_expired | listing_id | {reason} |

Retention: indefinite. These rows are the legal record of borrower
choice.

## Test plan

### Atomicity

- Concurrency test: 5 lenders POST claim against the same listing
  simultaneously. Exactly 3 must succeed. 2 must receive
  `claim_cap_reached`.

- Pick test: borrower POSTs pick. All `claimed` rows transition in one
  transaction. A retry of the same pick returns `listing_not_pickable`.

### Borrower ownership

- Borrower with cookie for deal A attempts pick on listing for deal B.
  Returns 403 `listing_not_owned_by_borrower`. No write occurs.

### RLS

- Anonymous PostgREST scrape of `marketplace_listings` returns 0 rows.
- Lender membership for bank X reads listing where X is in
  `matched_lender_bank_ids` — succeeds.
- Lender membership for bank X reads listing where X is NOT in
  matched — returns 0 rows.
- Brokerage ops membership reads all listings — succeeds.

### Scheduler

- Listing with `pending_preview` and past `preview_opens_at` flips on
  next tick. Idempotent.
- Listing with `claiming` and 0 claims at `claim_closes_at` flips to
  `expired`. Notifies ops.
